# Cloud Memo - n8n 웹훅 연동 명세서

**문서 버전:** v1.2  
**최종 수정일:** 2026-05-27  
**작성 목적:** Cloud Memo 앱과 n8n 자동화 워크플로우 간의 데이터 송수신 규격 및 처리 방식을 정의하여 n8n 워크플로우 개발자가 참고할 수 있도록 함.

---

## 1. 개요

Cloud Memo 애플리케이션은 사용자가 작성한 **텍스트, 이미지 파일(.jpg, .png 등) 및 음성 파일(.webm 등)**에 한정하여 n8n 워크플로우의 웹훅(Webhook) 트리거로 전송할 수 있습니다. 
전송 시 앱 내에 임시 상태(처리 중, `isProcessing: true`)의 메모를 생성하며, 최대 120초간 n8n의 동기식 응답(Synchronous Response)을 대기합니다. 
n8n 워크플로우가 처리를 완료하고 특정 규격의 JSON 응답을 반환하면, 앱은 그 결과를 받아 기존 메모를 최종 업데이트하고 사용자에게 결과를 보여줍니다.

---

## 2. 웹훅 요청 (Cloud Memo -> n8n)

### 2.1. 기본 정보
- **메서드:** `POST`
- **Content-Type:** `multipart/form-data`
- **헤더 (Headers):**
  - `X-N8N-TOKEN`: 보안 토큰 (앱 내 설정에서 토큰을 기입한 경우에만 포함됨. n8n 웹훅 노드에서 Header Auth 등으로 검증에 사용 가능)

### 2.2. 페이로드 파라미터 (Form Data)

| 필드명 | 타입 | 필수 여부 | 설명 |
| :--- | :--- | :---: | :--- |
| `title` | String | 필수 | 메모 내용의 첫 10글자 또는 음성/이미지 자동 생성으로 빌드된 제목 |
| `content` | String | 선택 | 사용자가 입력한 메모 내용 (음성/이미지 첨부 시 공백 허용. 단, FormData 키는 필수 포함) |
| `memoId` | String | 필수 | Firestore에 임시 생성된 메모의 고유 Document ID |
| `fileCount` | String | 선택 | 첨부된 파일의 총 개수 (이미지 + 음성 파일 합산 개수, 예: "2") |
| `file_{index}` | File | 선택 | 전송된 **이미지 또는 음성** 파일 데이터 (예: `file_0`, `file_1`...) |

> **참고:** 첨부된 모든 이미지 및 음성 파일은 Firestore 및 Storage에 저장되지 않으며, 오직 n8n 웹훅으로만 다이렉트 전송됩니다. 원본 음성/이미지 보관이 필요하다면 n8n 워크플로우 내에서 Google Drive 등에 수집하여 직접 저장해야 합니다.

## 3. 웹훅 응답 (n8n -> Cloud Memo)

n8n 워크플로우는 데이터 처리가 끝난 후 `Respond to Webhook` 노드(또는 Webhook 노드의 기본 응답 설정)를 사용하여 **반드시 JSON 형식**으로 응답해야 합니다.
앱은 HTTP Status Code `200 OK`를 성공의 기본 전제로 하며, 응답된 JSON 내부의 필드를 분석하여 최종 성공/실패 여부를 결정합니다.

### 3.1. 성공 시 응답 규격

- **HTTP Status:** `200`
- **Content-Type:** `application/json`

**성공 시 권장 JSON 구조:**
```json
{
  "success": true,
  "status": "success",
  "title": "n8n에서 가공한 새로운 제목 (선택)",
  "output": "n8n에서 AI 등을 거쳐 가공된 최종 메모 내용 (선택)"
}
```

**앱 내 처리 로직:**
- `title` 필드가 존재하면 기존 메모의 제목을 해당 값으로 덮어씁니다.
- `output` 필드가 존재하면 기존 메모의 내용을 해당 값으로 덮어씁니다.
- 메모의 상태를 `isProcessing: false`, `n8nStatus: 'success'`로 업데이트하여 사용자에게 완료를 알립니다.

### 3.2. 실패 시 응답 규격

- **HTTP Status:** `200` (에러 메시지를 앱에 안전하게 전달하기 위해 200 반환 후 내부 필드로 에러 명시 권장. HTTP 4xx, 5xx 반환 시에도 에러로 처리됨)
- **Content-Type:** `application/json`

**실패로 간주되는 JSON 구조 조건 (하나라도 만족 시 에러 처리):**
- `status` 필드 값이 `"error"`, `"fail"`, `"failed"` 중 하나인 경우
- `success` 필드 값이 `false`인 경우
- `error` 필드나 `errorMessage` 필드에 실제 에러 내용이 들어있는 경우

**실패 시 권장 JSON 구조:**
```json
{
  "success": false,
  "status": "error",
  "errorMessage": "AI 처리 중 API 키 한도가 초과되었습니다."
}
```

**에러 메시지 추출 우선순위 (앱이 사용자에게 보여줄 메시지):**
1. `error.message` (error가 객체인 경우)
2. `error` (error가 문자열인 경우)
3. `errorMessage`
4. `message`
5. `output`
6. `response.statusText` (HTTP 에러인 경우)

**앱 내 처리 로직:**
- 추출된 에러 메시지와 함께 사용자에게 실패 알림을 표시합니다.
- 메모의 상태를 `isProcessing: false`, `n8nStatus: 'error'`, `n8nError: <에러 메시지>`로 업데이트합니다.

## 4. 타임아웃 및 주의사항
 
 1. **타임아웃 (180초 제한):** 
-    - Cloud Memo 앱은 전송 후 최대 **120초(2분)** 까지만 응답을 대기합니다.
-    - 워크플로우의 실행 시간이 2분을 초과할 경우, 앱에서는 "n8n 응답 시간이 초과되었습니다"라는 에러로 간주하여 실패 처리합니다.
-    - *권장:* 무거운 작업(예: 대용량 이미지 처리, 복잡한 LLM 체인)의 경우 성능을 최적화하여 120초 이내에 `Respond to Webhook` 노드에 도달하도록 워크플로우를 설계해야 합니다.
+    - Cloud Memo 앱은 전송 후 최대 **180초(3분)** 까지만 응답을 대기하도록 타임아웃 규격을 강화하였습니다. (모바일 음성 파일 전송 및 STT 연산 시간 확보 목적)
+    - 워크플로우의 실행 시간이 3분을 초과할 경우, 앱에서는 "n8n 응답 시간이 초과되었습니다"라는 에러로 간주하여 실패 처리합니다.
+    - *권장:* 무거운 작업(예: 대용량 이미지 처리, 복잡한 LLM 체인)의 경우 성능을 최적화하여 180초 이내에 `Respond to Webhook` 노드에 도달하도록 워크플로우를 설계해야 합니다.
 
 2. **응답 노드 설정 (Respond to Webhook):**
    - n8n Webhook 노드의 `Respond` 설정을 `Using 'Respond to Webhook' Node`로 설정하고, 워크플로우의 맨 마지막이나 논리적 종료 시점에 반드시 `Respond to Webhook` 노드를 배치하여 JSON 본문을 구성해 주어야 합니다.
 
 3. **인증(Auth) 설정:**
    - 앱에서 "보안 토큰"을 입력했다면, n8n의 Webhook 노드 설정에서 **Authentication**을 `Header Auth`로 설정하고, Credential에 Name을 `X-N8N-TOKEN`, Value에 해당 토큰값을 설정해야 안전하게 요청을 받을 수 있습니다.
 
 4. **지원 파일 형식 제한 (이미지 및 음성 지원):** 
    - n8n 자동화 작성 시 첨부 가능한 파일은 **이미지 형식(.jpg, .jpeg, .png, .gif, .webp) 및 음성 파일(.webm, .m4a, .mp3, .wav)**로 제한됩니다.
    - **지원하지 않는 형식:** 영상 파일(MP4 등), 일반 문서 바이너리 파일(PDF, ZIP, DOCX 등) 및 기타 미정의 파일.
    - 음성 파일은 클라이언트 단에서 별도의 이미지 압축 처리를 거치지 않고 원본 파일 상태 그대로 `file_${index}` 필드에 삽입되어 전송됩니다.
-    - 음성은 이미지에 비해 용량이 크고 STT(음성인식) 가공 시간이 소요될 수 있으므로, n8n 워크플로우 응답이 120초 동기 대기 제한을 초과하지 않도록 최적화된 STT 파이프라인 설계를 권장합니다.

+## 5. 모바일 및 네트워크 안정성 강화 스펙 (IndexedDB & Wake Lock)
+
+Cloud Memo 앱은 음성 녹음 및 대용량 파일 전송 시 모바일 환경의 돌발 상황에 대응하기 위해 다음과 같은 전용 안정성 기술을 탑재하고 있습니다.
+
+### 5.1. Screen Wake Lock (화면 절전 차단)
+- **작동 시점:** 사용자가 음성을 녹음하는 도중, 그리고 n8n 웹훅 전송 시작부터 완료(또는 최종 실패/타임아웃) 시점까지 백그라운드 스레드 정지를 방지하기 위해 작동합니다.
+- **기대 효과:** 긴 녹음 중이나 대용량 전송 도중에 모바일 화면이 잠기거나 시스템 절전 모드로 진입하여 전송이 중단되는 현상을 물리적으로 완전 예방합니다.
+
+### 5.2. IndexedDB 기반 로컬 예비 백업 및 복구
+- **작동 시점:** 음성 녹음 완료 즉시 브라우저 로컬 데이터베이스인 **`IndexedDB`**에 이진 파일(Blob) 데이터를 임시 영구 캐싱 보존합니다.
+- **트랜잭션 생명주기:**
+  - **성공 시:** n8n 웹훅 전송이 완벽하게 성공(HTTP 200 및 `isFinalSuccess` 검증 통과)하면 로컬 IndexedDB의 백업본은 즉시 자동 삭제(Clean Cleanup)됩니다.
  - **실패 시:** 네트워크 장애, 브라우저 탭 비정상 종료, 혹은 n8n 처리 실패(Logical Error) 발생 시 백업 오디오는 로컬 DB에 **안전하게 유지**됩니다.
  - **복구 흐름:** 사용자가 웹훅 작성 페이지에 재접속하면 시스템이 미전송 로컬 백업을 자동 감지하여 상단에 **[복구하기/삭제]** 배너를 노출하며, 복구 버튼 클릭 시 실패했던 음성 파일이 100% 동일하게 첨부 상태로 재생성되어 재시도가 가능합니다.
